# Техническое задание (ТЗ): интеграция LLM (Anthropic Claude) в Telegram-бота

## 1. Цель

Доработать существующий Telegram-бот погоды на Python так, чтобы он стал полноценным AI-ассистентом:
- отвечал умно на любой пользовательский текст (кроме команд),
- учитывал контекст диалога,
- поддерживал потоковый вывод ответа (streaming),
- контролировал дневной бюджет расходов на LLM в размере `$2/день`,
- сохранял рабочие погодные команды (`/weather`, `/dailyon`, `/dailyoff`),
- имел модульную архитектуру, улучшенную надежность и базовые тесты.

## 2. Исходное состояние проекта

Текущий проект:
- `bot.py` — монолитный файл со всей логикой,
- `.env.example` — только `TELEGRAM_BOT_TOKEN`,
- `requirements.txt` — `python-telegram-bot`, `requests`, `python-dotenv`,
- `README.md` — документация по погодным функциям.

## 3. Объем работ

### 3.1. Интеграция Anthropic

- Использовать только провайдера Anthropic.
- Добавить клиент LLM и вызовы модели Claude.
- Ответы генерируются для любого текстового сообщения пользователя, если это не Telegram-команда.
- На старте использовать семейство `claude-3` (конкретная модель задается через `.env`).

### 3.2. Контекст диалога

- Вести историю диалога для каждого чата.
- Передавать в модель последние сообщения (скользящее окно контекста).
- Контекст хранить в памяти процесса (in-memory) как MVP.
- Ограничить окно контекста по количеству сообщений и/или приблизительной длине, чтобы контролировать стоимость и задержку.

### 3.3. Потоковая отправка ответа (streaming)

- Реализовать потоковый режим генерации ответа от LLM.
- В Telegram показывать поэтапное формирование ответа через редактирование одного сообщения.
- Добавить безопасную частоту обновлений (throttle), чтобы не превышать лимиты Telegram API.
- При невозможности stream-потока использовать fallback на обычный единый ответ.

### 3.4. Ограничение бюджета `$2/день`

- Ввести дневной лимит расходов LLM на уровне приложения: `$2.00` в сутки.
- Учет расходов вести по текущему дню (UTC или отдельно оговоренная TZ).
- При исчерпании лимита:
  - не вызывать LLM,
  - возвращать пользователю вежливое сообщение о достижении дневного лимита,
  - предлагать повторить позже (на следующий день).
- Учет стоимости строить по usage из ответа API (входные/выходные токены и тариф модели).
- Тарифы хранить в конфиге (env), чтобы можно было обновлять без изменения кода.

### 3.5. Язык ответов

- По умолчанию ассистент отвечает на языке пользователя (автоопределение по входному тексту).
- Поддерживать корректные ответы на русском и английском.
- Если запрос на перевод на английский присутствует, выполнять перевод.

### 3.6. Логирование

- Разрешено логировать тексты пользовательских сообщений и ответов модели.
- Добавить структурированные логи:
  - входящие сообщения,
  - параметры вызова модели,
  - usage и рассчитанная стоимость,
  - ошибки (timeout, rate limit, auth и прочие),
  - факты срабатывания лимита бюджета.
- Добавить базовую маскировку чувствительных данных (если появятся секреты в логах).

### 3.7. Обработка ошибок и единый стиль сообщений

- Ввести единый модуль/слой обработки ошибок LLM/API.
- Для пользователя — короткие дружелюбные сообщения без технических подробностей.
- Для логов — детальная причина и класс ошибки.

## 4. Архитектура и рефакторинг

Разделить монолит `bot.py` на модули.

Рекомендуемая структура:
- `bot.py` — точка входа, сборка `Application`, регистрация хендлеров.
- `handlers/commands.py` — `/start`, `/weather`, `/dailyon`, `/dailyoff`.
- `handlers/chat.py` — обработка обычных текстовых сообщений и stream-ответов LLM.
- `services/weather_service.py` — погода и рекомендации по одежде.
- `services/llm_service.py` — Anthropic клиент, формирование prompt/messages, stream, usage.
- `services/context_store.py` — хранение и управление контекстом чатов.
- `services/budget_service.py` — учет стоимости, дневной лимит.
- `core/config.py` — чтение и валидация env-конфига.
- `core/errors.py` — доменные исключения и маппинг ошибок.
- `core/logging_setup.py` — настройка логирования.

Допускается близкая эквивалентная структура при сохранении разделения ответственности.

## 5. Переход на async-сетевые вызовы

- Убрать блокирующие HTTP-вызовы из async-потока.
- Для погоды: заменить `requests` на async-клиент (`httpx`/`aiohttp`) или безопасный executor.
- Для LLM: использовать асинхронный вызов Anthropic API и streaming API.
- Обеспечить таймауты и retry-политику (минимально для временных сетевых сбоев).

## 6. Конфигурация через `.env`

Обязательные переменные:
- `TELEGRAM_BOT_TOKEN`
- `ANTHROPIC_API_KEY`
- `LLM_MODEL` (по умолчанию модель семейства `claude-3`)
- `LLM_TEMPERATURE`
- `LLM_MAX_TOKENS`
- `LLM_TIMEOUT_SECONDS`
- `DAILY_BUDGET_USD` (default: `2.0`)
- `BUDGET_TIMEZONE` (например, `UTC`)
- `CONTEXT_MAX_TURNS`
- `STREAM_EDIT_INTERVAL_MS`
- `LOG_LEVEL`

Дополнительные переменные (опционально, но рекомендовано):
- `MODEL_INPUT_COST_PER_1M_TOKENS_USD`
- `MODEL_OUTPUT_COST_PER_1M_TOKENS_USD`

Обновить `.env.example` с полным набором параметров и безопасными значениями по умолчанию.

## 7. Изменения в пользовательском поведении бота

- Команды продолжают работать как раньше.
- Любой текст (не команда) обрабатывается LLM и получает умный ответ.
- Если бюджет исчерпан — LLM-ответы временно недоступны, команды погоды остаются рабочими.
- При stream-ответе пользователь видит постепенное появление текста.

## 10. Нефункциональные требования

- Совместимость: Python `3.10+`.
- Надежность: при сбоях LLM бот не падает и продолжает обслуживать команды.
- Производительность: ответ пользователю начинается быстро (для streaming — первая порция в приемлемое время).
- Поддерживаемость: четкое разделение модулей, понятные имена, минимизация дублирования.
- Безопасность: секреты не захардкожены, читаются только из env.
